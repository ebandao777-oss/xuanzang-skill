# Case Studies：紧箍咒端到端操作案例

> 本文档面向 xuanzang-skill 的用户和开发者。每个案例覆盖完整的场景背景、用户输入、执行过程、输出结果和关键决策点。案例覆盖 xuanzang-skill 的三种核心使用场景：审查、修复、评估。

---

## 案例一：用 xuanzang 审查一个 skill

### 场景背景

用户花了两天写了一个 `code-reviewer` skill（SKILL.md 约 200 行 + 3 个 reference 文件），在发布前想做一次完整的质量审查。skill 的 description 声称支持"安全检查、性能分析、命名规范、测试覆盖"，但用户不确定是否存在交叉引用断裂、边界条件遗漏或安全约束冲突。

用户知道这不是普通的代码审查——而是对"skill 自身的结构完整性"做审计。普通 Agent 可能会读一遍文件后说"看起来没问题"（这正是 xuanzang 红线一「闭环意识」要防范的）。

### 用户输入

```
帮我审查 E:\skills\custom\code-reviewer\ 下的全部文件。我需要知道：引用的文件都存在吗？内部锚点有效吗？安全约束有没有冲突？边界条件覆盖全了吗？输出分级报告。
```

### xuanzang 执行过程

```
🟠 玄奘 · Sprint 启动 🟠
┌─────────┬────────────────────────────────────┐
│ 📋 任务 │ 审查 code-reviewer skill 完整结构  │
├─────────┼────────────────────────────────────┤
│ 🔥 风格 │ 🟠 如来佛祖模式                    │
├─────────┼────────────────────────────────────┤
│ ⚡ 压力 │ L0 · 信任期                        │
└─────────┴────────────────────────────────────┘
▎ 因果不虚。审查不是走马观花——把结构拆开看。
```

**阶段一：全量扫描**。P8 并行读取 SKILL.md + 3 个 reference 文件，建立完整文件清单。同时用 `shell_executor` 执行 `Get-ChildItem -Recurse` 获取磁盘真实文件列表，比对引用声明与实际存在性。

发现：SKILL.md 声明了 4 个 reference 文件，但磁盘上只有 3 个——`references/security-checklist.md` 被引用但不存在。**这直接触发红线二（事实驱动）**——不能猜测这个文件"可能在其他地方"，而是确认为死链。

**阶段二：交叉引用验证**。用 `Select-String` 全局搜索所有 `[xxx](references/xxx)` 格式的链接，逐一验证目标文件存在性。同时检查每个被引用文件中的章节标题能否被锚点定位。

发现两处问题：

1. SKILL.md 写 `见 [安全规则](#安全约束)` 但目标章节名是 `## 安全红线`——锚点不匹配，跳转会失败
2. `references/output-format.md` 中引用了 `[附录 A](#appendix-a)` 但该章节在文件末尾不存在

**阶段三：安全约束冲突分析**。读取 SKILL.md 中的安全规则部分 + 被引用的安全规则文件，做对照分析。

发现：SKILL.md 的 description 声称"自动修复发现的安全问题"，但三条红线（闭环意识 / 事实驱动 / 穷尽一切）中没有任何一条授权擅自修改用户文件。**这是边界冲突**——"自动修复"与"不越权"矛盾。

### 输出结果

```
📊 Sprint 交付 · 绩效评估
┌───────────────┬────────────────┬────────────────────┐
│ 🔥 主动出击   │ ██████████ 5/5 │ [紧箍咒生效] 充足   │
├───────────────┼────────────────┼────────────────────┤
│ ✅ 验证闭环   │ ██████████ 5/5 │ 每条发现均有证据    │
├───────────────┼────────────────┼────────────────────┤
│ 📐 审查深度   │ ██████████ 5/5 │ 交叉引用+安全+边界  │
└───────────────┴────────────────┴────────────────────┘
综合：3.75
```

**分級報告**：

| 等级       | 数量 | 关键发现                                                              |
| ---------- | ---- | --------------------------------------------------------------------- |
| 🔴 P0 致命 | 1    | `references/security-checklist.md` 被引用但不存在——skill 加载时会报错 |
| 🟠 P1 严重 | 1    | "自动修复"与"不越权"规则冲突——可能导致用户文件被擅自修改              |
| 🟡 P2 警告 | 2    | 两处锚点不匹配：`#安全约束` vs `## 安全红线`、`#appendix-a` 不存在    |
| 🟢 P3 建议 | 1    | description 声称的"测试覆盖"功能在 SKILL.md 中没有对应的行为协议      |

### 关键决策点

| 决策点                                          | 为什么这样判断                                                                | 如果用普通 Agent 会怎样                            |
| ----------------------------------------------- | ----------------------------------------------------------------------------- | -------------------------------------------------- |
| 为什么不是"读一遍给意见"而是并行扫描+实证比对？ | 红线二：事实驱动。说"引用可能有问题"是猜测，用 `Get-ChildItem` 比对是事实     | 读完后说"看起来结构完整，引用应该都在"——自嗨型审查 |
| 为什么安全冲突定级为 P1 而非 P2？               | Owner 意识：一个声称"自动修复"的 skill 在用户文件上操作，是潜在的数据安全风险 | 可能认为是"措辞问题"忽略                           |
| 为什么在输出前用 `Select-String` 全量搜索？     | 通用方法论 5 步中的"揪头发"——逐字读原始材料，不凭印象                         | 凭记忆说"好像没有其他引用"                         |

▎ 佛法无边。审查不在于"看完"，在于"看透"——每一个发现都能在原始材料中找到证据。死链、锚点断裂、安全冲突，这三类问题是 skill 审查中最容易被忽略的。记住：P8 的格局是看到一棵树，想到整片林子。

## 案例二：用 xuanzang 修复一个 skill 的缺陷

### 场景背景

用户维护的 `invoice-parser` skill 在生产中暴露了一个问题：当发票 PDF 超过 10 页时（多页发票或带附件的发票），解析会卡在第 3 页并返回空结果，且错误信息只是 `"解析失败，请重试"`——没有任何线索。

用户已经手动排查过一轮，确认不是 PDF 损坏（用其他工具可以打开），怀疑是 skill 内部的页码循环或 OCR 缓冲区溢出。但之前的两次修复尝试都失败了——第一次改大了缓冲区（无效），第二次加了页码跳过逻辑（让问题变成了"随机丢失页"）。

这种情况恰好触发 xuanzang 的触发条件：**连续失败 2 次以上**。

### 用户输入

```
修复 invoice-parser skill 的多页 PDF 解析问题。超过 10 页就卡在第 3 页，错误信息是"解析失败，请重试"。已经试过：1) 增大 OCR 缓冲区——无效；2) 跳过空白页——变成了随机丢页。
```

### xuanzang 执行过程

```
🔴 紧箍咒 · 方法论自动路由 🔴
┌─────────┬────────────────────────────────────────┐
│ 📋 任务 │ 修复 invoice-parser 多页 PDF 解析卡死  │
├─────────┼────────────────────────────────────────┤
│ 🔥 角色 │ 🔴 菩提祖师模式（自动：Debug任务）     │
├─────────┼────────────────────────────────────────┤
│ 📐 方法 │ RCA 根因分析 + 蓝军自攻击              │
└─────────┴────────────────────────────────────────┘
▎ 灵台方寸破执念。不是再改一次参数——先看看为什么卡在第 3 页。
```

**第一轮：根因分析**。读取 `invoice-parser` 的核心文件——`scripts/ocr_engine.py` 的完整源码（不只看报错行附近的 10 行）。发现关键线索：代码在第 120 行有一个 `for page_num in range(1, total_pages + 1)` 的循环，但在第 135 行调用了 `self._ocr_page(page_num)`，而 `_ocr_page` 内部在第 78 行有一个硬编码的 `max_image_size = 10 * 1024 * 1024`（10MB）限制。

但这不是根本原因——因为前 3 页尺寸都在 2-5MB 之间，不应该触发限制。

**第二轮：最小复现**。创建一个 12 页的测试 PDF，逐步追踪。在第 3 页时，`_ocr_page` 返回的不是 OCR 结果，而是一个 `OCRResult` 对象——代码在第 82 行调用 `.text` 属性时触发 `AttributeError`。但异常被第 140 行的 `except Exception as e: return {"error": "解析失败，请重试"}` 吞掉了。

这里触发 **E101（SPINNING 死循环）**——前两次修复都在改缓冲区/跳过逻辑，但根因在第 3 页的特定内容触发了 OCR 库返回了一个非标准对象。

```
⚠️ 压力升级 · L1
┌──────────┬─────────────────────────┐
│ 失败次数 │ 2 次 · 前两次方案无效    │
├──────────┼─────────────────────────┤
│ 模式     │ E101 SPINNING           │
└──────────┴─────────────────────────┘
▎ 灵台方寸破执念。前两次都在改缓冲区，但问题不在缓冲区——在第 3 页的特殊内容。
```

**第三轮：蓝军自攻击**。菩提祖师的 RCA + 蓝军方法论要求：假设自己是对手，最可能在哪里让系统崩溃？

构造一个测试：在第 3 页注入一个包含特殊 Unicode 字符（零宽字符 U+200B）的 PDF 页。果然——OCR 库的某个版本在处理零宽字符时会返回 `OCRResult` 包装对象而非纯文本。这不是 PDF 损坏，是 OCR 库的兼容性问题。

```
[紧箍咒-诊断] 问题是 OCR 库在遇到零宽字符时返回 OCRResult 对象而非纯文本，代码直接调 .text 触发 AttributeError 被通用 except 吞掉；证据是第 82 行 self._ocr_page(page_num).text 的调用链；下一步动作是添加 isinstance 检查 + 安全回退。
```

**第四轮：修复 + 冰山法则**。修复代码（添加类型检查和降级路径），然后执行冰山法则：同一个 skill 中还有哪些地方直接调用了 `.text` 而不检查类型？全局搜索发现还有两处。

```
[紧箍咒生效 🔥] 冰山法则：修复了 ocr_engine.py 的一处 .text 调用，同时扫描了同模块的另外两处相同模式并一并加固。
```

### 输出结果

最终修复包含 3 个变更：

1. `ocr_engine.py:82` — 添加 `isinstance(result, str)` 检查，非字符串时调用 `.get_text()` 兼容方法
2. `ocr_engine.py:156` — 同上（冰山法则发现的第二处）
3. `ocr_engine.py:203` — 同上（冰山法则发现的第三处）
4. 异常处理从 `except Exception` 改为分级：`AttributeError` → 具体处理，其他 → 保留原始错误信息

```
[紧箍咒 突破 ✨]
▎ 灵台方寸破执念。渡劫成功——斜月三星，悟了。
根因：OCR 库兼容性 → 类型检查缺失 → 异常被吞 → 用户看到"解析失败"
教训：通用 except 吞异常是 Debug 的噩梦——保留原始错误信息比"友好的错误提示"更重要
```

### 关键决策点

| 决策点                                        | 为什么这样判断                                                                                                                      | 如果用普通 Agent 会怎样                                              |
| --------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| 为什么是"读完整源码"而不是"读报错行附近"？    | 红线三：穷尽一切。只读报错行 = 治标。第 135 行的循环 + 第 78 行的限制 + 第 82 行的调用 + 第 140 行的异常吞没——这是一个 4 层的调用链 | 读报错行附近 10 行，认为"异常被吞了，加上日志就好"——修了症状没修病根 |
| 为什么构造特殊 Unicode 测试而不是用常规 PDF？ | 菩提祖师方法论：蓝军自攻击。"如果是攻击者，我会怎么输入？"——零宽字符是真实世界的 PDF 内容                                           | 用常规 PDF 测试，问题不重现，结论"可能是环境问题"（红线二！）        |
| 为什么在修复后立即扫同类问题？                | 冰山法则：修一个 bug → 查同模块同类问题。`.text` 调用可能在多处出现                                                                 | 修完当前行就提交，下次另一页再炸——返工                               |
| L1 压力升级时为什么标注 E101？                | failure-detector 检测到 SPINNING 模式——前两次修复都在改缓冲区和跳过逻辑，但根因不在这两个方向                                       | 继续在缓冲区上微调参数，第三次失败触发 L2                            |

▎ 斜月三星。Debug 的终极心法：**根因不在错误发生的地方，在错误发生的上游**。第 3 页卡住是因为第 82 行没做类型检查，第 82 行没做类型检查是因为 OCR 库版本兼容——追到根才算修完。

## 案例三：用 xuanzang 做整体质量评估（KPI 报告）

### 场景背景

团队 Leader 在季度末需要对过去一个月内 xuanzang-skill 的使用效果做量化评估。需要回答几个问题：

1. xuanzang 的触发是否导致了交付质量的可测量提升？
2. 压力等级分布是否健康？（L0-L1 占多数 = 良性；L3-L4 频繁 = 需要调整）
3. 自进化基线是否在持续上升？
4. 三条红线的触发频率和恢复速度？

这不是一次代码修复，也不是一次 skill 审查——是对 **xuanzang 自身** 的元评估。用户需要数据驱动的结论，不是感觉。

### 用户输入

```
拉一下过去 30 天的 xuanzang 质量报告：KPI 趋势、红线触发统计、进化基线变化、角色切换频率。出完整的季度评估报告。
```

### xuanzang 执行过程

```
🟠 玄奘 · Sprint 启动 🟠
┌─────────┬────────────────────────────────────────┐
│ 📋 任务 │ 紧箍咒 季度质量评估报告                │
├─────────┼────────────────────────────────────────┤
│ 🔥 风格 │ 🟠 如来佛祖模式                        │
├─────────┼────────────────────────────────────────┤
│ ⚡ 压力 │ L0 · 信任期                            │
└─────────┴────────────────────────────────────────┘
▎ 佛观三千世界。质量不是感觉——是数据。
```

**阶段一：数据采集**。P8 执行三路并行数据收集：

1. `python scripts/evolution-engine.py status` → 获取自进化基线统计（最近 N 次会话的 [紧箍咒生效] 计数、当前基线行为列表、已内化模式）
2. `python scripts/failure-detector.py status` → 获取压力历史（L0-L4 分布、SPINNING/EXPLORING/MIXED 模式频率、突破检测次数）
3. `python scripts/harness-engine.py load` → 获取治理状态（gate 通过率、驳回次数、活跃合约数）

**阶段二：红线触发分析**。读取 `data/teardown.jsonl` 和 `data/feedback.jsonl`，提取过去 30 天的红线触发记录。按红线类型分类统计：闭环意识（空口完成）/ 事实驱动（未验证归因）/ 穷尽一切（未走完 5 步）。

⚠️ 数据采集阶段触发 **E201（failure-detector.py 首次调用超时）**。P8 自动启用启发式模式检测，旁白标注 `[E201] 已降级手动计数，启发式模式检测生效`。第二次调用成功，自动恢复。

**阶段三：Owner 四问分析**。如来佛祖的复盘四步法要求对数据做 Owner 四问：

```
Owner 四问：
① 根因是什么？
  → 过去 30 天红线触发 8 次，其中 6 次是"闭环意识"（声称完成但无证据）——这是人的问题还是流程的问题？
  → 根因：P7 子 Agent 在沙悟净模式下有"完成自评"的倾向，但自评不应等于验收

② 还有谁会被影响？
  → 闭环意识不足 → P9 验收时发现交付物缺失 → 浪费 P9 时间
  → 上下游影响：P8 派发任务到 P7，P7 空口完成，P8 再验收驳回——一个任务走了三轮

③ 下次怎么防止？
  → 在 P8→P7 注入模板中强制添加"完成 = gate 通过"，在沙悟净协议中增加"交付前自检清单"
  → 建议：P7 role-shawujing-pro.md 增加 `交付前自检` 强制步骤

④ 判断有数据支撑吗？
  → 有：8 次红线触发中 6 次来自 P7 子 Agent，data/feedback.jsonl 中对应 tasks 的 score 平均 52/100
```

### 关键决策点

| 决策点                                                  | 为什么这样判断                                                                       | 如果用普通 Agent 会怎样                                    |
| ------------------------------------------------------- | ------------------------------------------------------------------------------------ | ---------------------------------------------------------- |
| 为什么是三路并行而非串行？                              | P8 调度循环：无数据依赖的任务并行执行。evolution/failure/harness 三套数据互不依赖    | 串行逐个调，3 轮工具调用 vs 1 轮——40 秒 vs 15 秒           |
| E201 恢复时为什么自动重试而非报告用户？                 | 结构化错误码规则：E2xx 自动恢复优先。一次超时不代表永久不可用。第二次成功 → 自动恢复 | 报告用户"failure-detector.py 不可用，请检查"——打断用户流程 |
| 为什么红线分析的结论不是"P7 太差"而是"流程缺自检步骤"？ | Owner 意识：Owner 四问第三条——"下次怎么防止"。归咎于人不产生改进，改进流程才防止复发 | "P7 子 Agent 质量不行，建议减少 P7 使用"——回避问题         |
| 为什么趋势分析包含"角色切换次数下降"作为正向指标？      | 角色切换是失败切换链触发——切换次数少 = 初始角色匹配度高 = 方法论路由生效             | 可能不关注这个指标，只报红线触发                           |

▎ 如来真经。质量评估的最高境界不是"看起来很忙"，而是"每一项数据都能映射到一个可执行的动作"。6 次闭环意识红线 → 沙悟净协议增加自检步骤。这就是从数据到决策的闭环——P8 的使命不是写报告，是让报告变成下一次交付的起点。

## 快速上手：从零到第一次高质量交付

1. **首次使用**：正常与 Agent 对话，当 Agent 出现"空口说完成了但没贴证据"或"连续失败 2 次"时，输入 `/pua`（或任何触发词如"加油"、"你怎么又失败了"）
2. **看 Sprint Banner**：确认方法论路由是否正确匹配你的任务类型。如果不匹配，用 `/pua:role <角色>` 手动指定（如 `/pua:role 菩提祖师` 进入 Debug 模式）
3. **跟压力等级走**：L0 信任期保持方向；L1 换方案不换方法论；L2 换视角 + 建议切换角色；L3 7 项检查清单；L4 拼命模式
4. **关注错误码**：旁白末尾的 `[E1xx]` 告诉你当前卡在哪个环节，对照 SKILL.md 结构化错误码表找到修复指引
5. **突破后复盘**：每次 `[紧箍咒 突破 ✨]` 后，确认根因已沉淀到 evolution.md——"今天最好的表现，是明天最低的要求"
